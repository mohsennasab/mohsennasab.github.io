# Editing this site, end to end

A practical tour: how the project is laid out, what each file does, and how to make every kind of common edit. Windows / PowerShell oriented.

---

## 1. Run the site, build the site

Open PowerShell (a fresh window, not one that was open before Node was installed):

```powershell
cd "c:\OneDrive\OneDrive - AECOM\Automation Projects\Website"
npm run dev          # development server at http://localhost:4321
                     # every save reloads the browser within ~1 second
                     # Ctrl+C in the terminal stops it
```

Other commands:

```powershell
npm run build        # production build into dist\
npm run preview      # serve the production build locally, to double-check
npm install          # only needed after pulling in new dependencies
```

If `npm` is "not recognized": close the PowerShell window and open a new one. Node is on your User PATH but PowerShell sessions read PATH at startup.

## 2. The 30-second tour of an Astro file

Almost every page and component on this site is a `.astro` file. They have three parts:

```astro
---
// 1. Frontmatter. JavaScript/TypeScript. Runs at build time only.
//    Use it for imports, data, and logic.
import Base from '../layouts/Base.astro';
const courses = ['ENGR467', 'ENGR368'];
---

<!-- 2. Template. HTML with {expressions} in curly braces. -->
<Base title="Teaching">
  <h1>Courses</h1>
  <ul>
    {courses.map((c) => <li>{c}</li>)}
  </ul>
</Base>

<style>
  /* 3. Optional scoped CSS. Only affects this file's elements. */
  h1 { color: var(--accent); }
</style>
```

That is it. Variables come from the top, you sprinkle them in the middle with `{ }`, you can include other components like HTML tags. No build step to think about while you edit; `npm run dev` rebuilds on save.

## 3. Folder structure at a glance

```
Website\
  EDITING.md                  ŌåÉ you are here
  README.md                   ŌåÉ project overview, deploy steps
  package.json                ŌåÉ dependency list and run scripts
  astro.config.mjs            ŌåÉ Astro configuration (site URL, integrations)
  tsconfig.json               ŌåÉ TypeScript settings
  .gitignore                  ŌåÉ what NOT to commit (node_modules, dist, etc.)
  _convert_images.py          ŌåÉ regenerates web-sized photos from originals

  Mohsen\                     ŌåÉ original headshots, podcast photo, teaching screenshots (gitignored)
  Photos\                     ŌåÉ original phone photos: JPGs and HEICs (gitignored)

  public\                     ŌåÉ copied verbatim to the site root
    CNAME                       (the custom domain for GitHub Pages)
    favicon.svg
    robots.txt
  .github\workflows\deploy.yml ŌåÉ GitHub Actions: builds and publishes on push

  node_modules\               ŌåÉ installed packages (gitignored, autogenerated)
  dist\                       ŌåÉ built site (gitignored, regenerated by npm run build)
  .astro\                     ŌåÉ Astro's internal cache (gitignored, autogenerated)

  src\                        ŌåÉ everything we actually wrote
    consts.ts                   site name, nav, social URLs, YouTube links, GitHub username
    styles\global.css           the whole design system (colors, fonts, spacing, components)
    layouts\Base.astro          page shell: <head>, nav, footer, theme toggle
    components\                 reusable building blocks (Nav, Footer, Hero, Card, etc.)
    pages\                      every file here becomes a URL
    assets\                     web-sized images (headshot, portraits, gallery photos)
```

The only thing that ever ships to GitHub is `src\`, `public\`, the config files at the root, `package.json`, `package-lock.json`, the workflow, and the docs. `node_modules\`, `dist\`, `.astro\`, `Mohsen\`, and `Photos\` are gitignored.

## 4. Pages: every file in src\pages\ is a URL

Astro routing is "file = URL". No router config to maintain.

| File | Becomes |
| --- | --- |
| `src\pages\index.astro` | `/` |
| `src\pages\publications.astro` | `/publications` |
| `src\pages\projects.astro` | `/projects` |
| `src\pages\tools.astro` | `/tools` |
| `src\pages\teaching.astro` | `/teaching` |
| `src\pages\photography.astro` | `/photography` |
| `src\pages\media.astro` | `/media` |
| `src\pages\404.astro` | the custom not-found page |
| `src\pages\blog\index.astro` | `/blog` |
| `src\pages\tutorials\index.astro` | `/tutorials` (overview) |
| `src\pages\tutorials\python-fundamentals.astro` | `/tutorials/python-fundamentals` (sub-page) |

To **add a new top-level page**, create `src\pages\my-new-page.astro` and it lives at `/my-new-page`. To **add a sub-page**, make a folder: `src\pages\tools\arc14.astro` becomes `/tools/arc14`.

To **show the new page in the nav bar**, also add it to the `NAV` array in `src\consts.ts` (see Section 6).

## 5. Layouts and components: where the shared parts come from

```
src\
  layouts\
    Base.astro          The page shell every page wraps itself in. Owns the <head>
                        (title, meta, fonts), the Nav, the Footer, the dark/light
                        theme toggle, and Astro's view transitions.

  components\
    Nav.astro           Sticky top navigation bar. Reads the NAV array from consts.ts.
    Footer.astro        Site footer with links and social icons.
    Hero.astro          The big homepage hero (name, role, bio, headshot, CTAs).
                        Used only by src\pages\index.astro.
    PageHeader.astro    The eyebrow + title + lede block at the top of every non-home
                        page, with an optional faded photo background.
    SectionHead.astro   Smaller section heading: eyebrow + h2 (or h3) + lede.
    Card.astro          A generic linked card (title, index label, description, hover).
    Socials.astro       The social-icon row used in the hero and the footer.
```

When you open any page file (say `src\pages\teaching.astro`), the pattern is always the same:

```astro
---
import Base from '../layouts/Base.astro';
import PageHeader from '../components/PageHeader.astro';
import SectionHead from '../components/SectionHead.astro';
import headerBg from '../assets/photos/photo-25.jpg';

const courses = [ /* the data for this page */ ];
---

<Base title="Teaching">                          <!-- wraps the page in the shell -->
  <PageHeader eyebrow="// teach" title="..."
              lede="..." bgImage={headerBg} />   <!-- page header -->

  <section class="section">                      <!-- one section of content -->
    <div class="wrap">
      <SectionHead eyebrow="// courses" title="Courses taught" />
      <!-- markup that uses the data from frontmatter -->
    </div>
  </section>

  <!-- more sections -->
</Base>
```

Once you see this pattern in one file, every page reads the same way.

## 6. src\consts.ts: the site-wide knobs

Open [src\consts.ts](src/consts.ts). This is the single place to edit:

- **Your name, role, tagline, email** (`SITE` object). Anything here flows into the `<title>` tag, the social-share meta, the footer, and so on.
- **The navigation bar** (`NAV` array). The order in this array is the order in the nav. To remove a page from the nav (without deleting the page itself), delete its line here.
- **Your social profile URLs** (`SOCIALS` array). To add a new one (Twitter, Mastodon, ORCID), add an entry and add a matching icon in `src\components\Socials.astro` (the `iconSvg` function near the top).
- **Your GitHub username** (`GITHUB_USER`). Used by the Tools page to build repo links.
- **Your YouTube channel and playlist URLs** (`YT` object). Used by Teaching and Tutorials.

Changing `consts.ts` updates the whole site at once.

## 7. src\styles\global.css: colors, fonts, spacing

[src\styles\global.css](src/styles/global.css) is the design system. The top of the file is a list of CSS variables you can edit to retheme the whole site:

```css
:root {
  --bg:          #232f3a;     /* page background */
  --bg-elev:     #2e3d49;     /* cards, raised surfaces */
  --bg-elev-2:   #38495a;     /* footer, strongest surface */
  --border:      #45596c;
  --text:        #eef4fb;     /* body text */
  --text-dim:    #b6c7d9;     /* secondary text */
  --text-faint:  #8099ad;     /* meta labels */
  --accent:      #88BDF2;     /* the "Stormy Morning" sky blue */
  --accent-2:    #BDDDFC;     /* the lighter sky blue */
  ...
  --font-sans:   'Inter Variable', ...;
  --radius:      16px;
  --maxw:        1080px;
  --nav-h:       62px;
}

[data-theme="light"] {
  /* the same variables, but for light mode */
  --bg:    #f3f7fb;
  --text:  #283643;
  ...
}
```

Want a different accent color sitewide? Change `--accent` and `--accent-2` here, save, the whole site updates. Want wider pages? Bump `--maxw`. Want bigger gaps? Use the editor's search to find `clamp(2.8rem, 6vw, 4.8rem)` in `.section` and tweak.

Below the variables, `global.css` defines reusable classes (`.wrap`, `.section`, `.card`, `.chip`, `.btn`, `.eyebrow`, `.entry-list`, `.timeline`, `.masonry`, `.placeholder-note`). When you see `class="btn btn--primary"` or `class="card"` in a page, that style is in `global.css`.

## 8. Common edits, by example

### a. Change a piece of text on a page

Open the page file in `src\pages\`. Most prose is right there in the template (between the `<Base>` tags), and most lists are in `const` arrays at the top of the frontmatter (between the `---` fences).

Example, to fix a typo in your bio: open `src\components\Hero.astro` and edit the `<p class="bio">...</p>` paragraph.

Example, to add a course to the teaching page: open `src\pages\teaching.astro`, find the `stthomas` array, and add a new line:

```ts
{ code: 'ENGR499', title: 'Independent Study', desc: 'One-line description.' },
```

Save. The browser reloads.

### b. Add a new section to a page

Add a new `<section>` block inside the page's `<Base>`:

```astro
<section class="section">
  <div class="wrap">
    <SectionHead eyebrow="// new thing" title="My new section"
                 lede="Optional one-line description." />
    <p class="prose">Whatever you want to say.</p>
  </div>
</section>
```

The `class="section"` gives it consistent vertical spacing. The `class="wrap"` keeps it inside the page's content width.

### c. Reorder sections

`<section>` blocks just render top-to-bottom. To move a section up, cut it and paste it higher in the file. Save.

### d. Change which photo a page header uses

Open the page file. Near the top, find:

```ts
import headerBg from '../assets/photos/photo-15.jpg';
```

Change the number to any photo in `src\assets\photos\`. Save.

The home hero is the same idea but in `src\components\Hero.astro` (look for `heroBg`).

### e. Move the focal point of a header photo

In the page file, in the `<PageHeader ... />` call, change `bgPos`:

```astro
<PageHeader ... bgImage={headerBg} bgPos="center 60%" />
```

`0%` = top of the photo, `50%` = middle, `100%` = bottom. Same idea horizontally for the first number.

### f. Change every header's photo opacity

Open `src\components\PageHeader.astro`. In the `<style>` block:

```css
.ph-bg { ... opacity: 0.32; ... }
[data-theme='light'] .ph-bg { opacity: 0.4; }
```

Lower = more faded, higher = more visible photo. The two values are dark and light mode.

The `/tutorials` page has its own header layout (not the shared `PageHeader`), so its opacity lives in `src\pages\tutorials\index.astro` under `.t-bg`. The home hero is in `src\components\Hero.astro` under `.hero-bg`.

### g. Add a new page and show it in the nav

1. Create `src\pages\talks.astro`. Copy the structure of any existing page and rewrite the content.
2. Open `src\consts.ts`, find the `NAV` array, add `{ label: 'Talks', href: '/talks' },` where you want it in the order.
3. Save. The dev server picks it up.

### h. Replace the headshot or a portrait

Open File Explorer to `src\assets\`. The relevant files are `headshot.jpg`, `portrait-1.jpg`, `portrait-2.jpg`, `podcast.jpg`, `teaching.jpg`, `teaching2.jpg`. Overwrite any of them with a new JPG (similar size, around 1500 pixels on the long edge). Save. Refresh the browser.

If you want to regenerate them from the originals in `Mohsen\` instead, edit `_convert_images.py` (the bottom of the script lists which Mohsen file becomes which web file) and run `python _convert_images.py`.

### i. Add or remove a gallery photo

Add: drop any JPG into `src\assets\photos\` named `photo-40.jpg` (or any filename). It is in the gallery automatically. About 1800 pixels on the long edge is the right size.

Remove: delete the file from `src\assets\photos\`. Gone.

For a "from-scratch regenerate" workflow (handles HEIC files and resizing for you): put the original in `Photos\`, then run

```powershell
python _convert_images.py
```

That rewrites everything in `src\assets\photos\photo-NN.jpg` in date order.

### j. Change a link or button label

These are just plain text in the templates. Use VS Code's search (Ctrl+Shift+F) to find the label, edit it, save.

### k. Change the color theme

Edit the `:root` variables in `src\styles\global.css`. Want green accents? Change `--accent` to `#7fb573` (or any other color), save. The whole site is in green. Same idea for `--bg`, `--text`, etc.

### l. Change the font

The site uses Inter through the npm package `@fontsource-variable/inter`, imported in `src\layouts\Base.astro`. To switch to a different Fontsource font (say, back to Hubot Sans):

```powershell
npm install @fontsource-variable/hubot-sans
```

then in `src\layouts\Base.astro` change the import:

```ts
import '@fontsource-variable/hubot-sans';
```

and in `src\styles\global.css` change `--font-sans` to start with `'Hubot Sans Variable'`. Browse [fontsource.org](https://fontsource.org) for other variable fonts; the package name and the family name (the `... Variable` string) are listed on each font's page.

## 10. The public folder: drop-in HTML / files

Anything you put in `public\` gets served at the site root with the same path.

- `public\favicon.svg` lives at `/favicon.svg`.
- `public\CNAME` is the custom-domain marker GitHub Pages uses.
- `public\apps\streamscan\index.html` would be live at `/apps/streamscan/`.

This is how your future standalone HTML apps and dashboards get hosted: drop them in, link to them from any page.

## 11. Deploying (when you are ready)

The full deploy steps are in [README.md](README.md). Short version:

1. Push the project to a GitHub repo (recommended name `<your-username>.github.io`).
2. In GitHub: Settings ŌåÆ Pages ŌåÆ Source ŌåÆ "GitHub Actions". The workflow `.github\workflows\deploy.yml` builds and publishes on every push to `main`.
3. In GitHub: Settings ŌåÆ Pages ŌåÆ Custom domain ŌåÆ `hydromohsen.com`.
4. In Squarespace DNS: add the four A records to GitHub's IPs, the `www` CNAME, and remove the old Google Sites records. (README has the exact list.)
5. Tick "Enforce HTTPS" once DNS propagates.

## 12. Things you cannot break by accident

- Editing photos: worst case the page looks off, you swap back.
- Editing a `.astro` file: if you make a typo, the dev terminal prints a clear error pointing to the line. Fix, save, it reloads.
- Re-running `_convert_images.py`: only writes into `src\assets\`. Inputs (`Mohsen\`, `Photos\`) are untouched.
- `npm run dev` and `npm run build` only touch the project folder.

If you ever wedge yourself badly, once this is in git you can run `git status` to see what changed and `git checkout -- <file>` to throw away the changes.
